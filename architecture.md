# プロジェクト設計仕様書 (再定義版)

## 1. プロジェクトの目的
本リポジトリは、**「AIによる保守・開発を前提としたWebユーティリティ・スイート」**です。
- **Type A (ポータブル):** HTML1枚完結。ビルド不要。
- **Type B (コンポーネント):** TS/React/Vite。ビルド必須。
- **原則:** AI-Friendlyであり、外部依存を最小限に抑えること。

## 2. 開発・運用ルール (AIへの指示用)
本プロジェクトの保守・開発時には、以下のログ出力形式を厳守する。
- **history.json 追記用:** インデントを維持し、末尾にカンマを付与したJSON形式。
- **README更新:** 変更履歴テーブルを必ず追記する。

## 3. 主要な技術スタックと構成
- **フロントエンド:** HTML, JS (Vanillaメイン), Vite, TypeScript (一部)
- **バックエンド/同期:** Firebase (Realtime Database / Firestore)
- **インフラ:** GitHub Pages
- **開発ツール:** Streamlit (高負荷計算用)

## 4. プロジェクトの経緯と設計判断 (History Summary)
- **初期化:** `landmine` プロジェクトから「メタデータ作成・履歴記録・README更新」のセット運用を徹底。
- **技術的判断:** 
    - `airhockey` ではTypeScript/Viteで開発を開始したが、ROI（費用対効果）を考慮し、JS直接編集の運用へ移行。
    - `landmine` では、Firebaseの `onValue` 監視における競合を「ガード節」で解決する方針を採用。
    - `text-layouter` では、AI認識阻害と人間可読性の両立のため、グリッドエンジンを自作。
- **運用上の教訓:** 
    - 複雑な関数分割よりも、状態ごとのロジック分離を優先する。
    - 開発ログ（history.json）はAIの状況把握の命綱であるため、これを読み込ませることで「ハルシネーション（捏造）」を抑制する。

## 5. 現在の「正しい」開発フロー
1. **コンテキストの固定:** 質問前に必ず `history.json` と `README.md` を読み込ませる。
2. **根拠の要求:** AIに対し「この回答は `history.json` のどの記述に基づいているか？」と根拠を問う。
3. **仕様の更新:** 修正や機能追加のたびに、AIに `history.json` への追記内容を出力させ、それを手動で保存する。
