# Landmine Game (Realtime Multiplayer)

多人数リアルタイム通信を用いた地雷ゲーム。

## 設計方針
- **Lobby:** Firebase Realtime Database を使用し、ホストの動的委譲と参加者同期を行う。
- **Game:** `landmine.html` にてゲームロジックを実行。ホストが開始ボタンを押すことで全員の画面が同期する。
- **Security:** Firebase Database Rules にて、ルーム参加者のみがデータを読み書きできるように制限。

## 開発運用ルール
- ログ出力形式（history.json・README更新テーブル）を遵守すること。



## 技術的要点と注意点

### ロビー機能およびホスト委譲ロジックについて
ロビーおよびホスト委譲の実装において、以下の点に注意して設計を行っています。

*   **`onDisconnect` の制限と回避:** 
    `onDisconnect` は `set` や `remove` などの単純な操作のみをサポートしています。`transaction` を直接メソッドとして呼び出すことはできないため、離脱時の状態更新は「単純な削除予約」と、残ったメンバーによる `onValue` 監視・立候補という「検知→立候補」の2段階構成で実装しています。
*   **非同期競合の回避:**
    Firebaseのキャッシュとサーバー側の状態不一致を防ぐため、`snapshot.exists()` による存在チェックと、データが `null` の場合の強制リダイレクト処理を徹底し、セッションの不整合を回避しています。
*   **名前情報の永続化:** 
    ホストが部屋作成後にページ遷移が発生しても、`sessionStorage` を用いて名前情報を保持することで、ユーザーに再入力を求めないスムーズな参加フローを実現しています。
*   **ホスト選出の堅牢性:** 
    特定のホストを維持するのではなく、「ホストIDが空である」という状態を全員で監視し、空であれば即座に立候補（`runTransaction`）する「立候補制」を採用しました。これにより、ホストの離脱理由（通信切断・ブラウザ終了・リロード等）に関わらず、確実に新しいホストが選出されます。

2. データ設計の要約（README.md 等への記録用）
今回の仕様に基づき、Firebase Realtime Database のデータ構造を以下のように定義します。

rooms/{roomId}/game/grid:
x_y (string): 各マスの状態管理。
status: “hidden” (未開放) / “open” (開放済)。
mineType: null / “large” / “small” (地雷の種類)。
ownerId: 地雷の設置者ID。
explodedBy: 地雷を踏んだプレイヤーID。
rooms/{roomId}/game/players:
points: 初期値 50pts（※仕様に合わせて調整可能）。
minesRemaining: 大1, 小2 の残数管理。
rooms/{roomId}/game/logs:
誰が誰の地雷を踏んだかの履歴を配列として保持。
rooms/{roomId}/game/status:
“placing” (設置フェーズ) → “playing” (進行フェーズ) → “finished” (終了) の状態遷移。


### ロビーからゲーム画面への遷移とデータ引き継ぎ
Firebaseの `onDisconnect` を使用している場合、ページ遷移の瞬間にクライアントが「切断」とみなされ、プレイヤーデータが削除される問題が発生します。これを解決するために以下の設計を採用しています。

*   **データの永続化（スナップショット）:**
    遷移直前に、ロビーの不安定なプレイヤーリスト（`rooms/{roomId}/players`）を、ゲーム専用の永続パス（`rooms/{roomId}/game/players`）へコピー（`set`）します。
*   **遷移の確実性:**
    `landmine.html`（ゲーム画面）は、ロビーのデータではなく、この「ゲーム開始時に固定されたパス」を参照することで、遷移後のデータ消失を防ぎます。
*   **AIへのアドバイス（遷移設計の鉄則）:**
    *   **「ロビーのデータ＝一時的なもの」と割り切る:** ページ遷移を伴う構成の場合、ロビーのデータを直接ゲーム画面で参照してはいけません。
    *   **「開始時の状態」を確定させる:** ゲーム開始ボタンが押された瞬間のプレイヤーリストを「ゲーム開始時のスナップショット」として別パスに書き出す処理を必ず挟んでください。
    *   **URLパラメータの併用:** 誰が遷移したかを追跡するため、`?room=xxx&uid=yyy` のようにURLパラメータでユーザーIDを渡すことで、遷移先で即座に自分自身を特定できるように設計してください。


【ロビーhtmlからゲームhtmlへ遷移した際、毎回陥るデータが継承されずにゲームが動かない問題に対するプロンプト】
「ロビーからゲーム画面への遷移を実装します。ページ遷移による onDisconnect の発火でプレイヤーデータが消えないよう、ゲーム開始時に rooms/{roomId}/game/players へデータをコピーし、遷移先ではその永続パスを参照する設計で実装してください。」

### 技術的知見 (Lessons Learned)
今回の開発で得られた、リアルタイム同期アプリにおける重要な設計指針です。

1.  **観測者と実行者の分離**:
    `onValue` 内で状態を更新 (`update`) する場合、その直後に `return` を記述してそれ以降のUI描画処理をスキップしないと、古いデータに基づいた描画が先行し、画面更新の不整合や遅延が発生する。
2.  **状態遷移のガード節**:
    `終了判定` → `開始更新処理` → `ゲーム中` → `設置中` という優先順位で処理を分岐させる構造が、競合を避けるための黄金律である。
3.  **サーバー主導のUI制御**:
    ローカル変数による状態管理は通信ラグによる同期ズレを招きやすい。ボタンの活性状態などは、可能な限り Firebase 上の最新データ（`grid` や `ready`）を元に算出することで、堅牢なUXを実現できる。
*   **0.1.8: リファクタリング方針の策定**
    *   `onValue` の肥大化に対し、過度な関数分割による同期バグのリスクを避けるため、現時点では「ガード節による状態分離」を優先する方針を決定。
    *   リファクタリングは機能追加のタイミングで、状態ごとの描画関数（`renderPlaying`, `renderWaiting` 等）へ段階的に切り出す計画とする。
