## 保守・管理情報 (Project Maintenance)

本リポジトリは「GitHub Pages」「Streamlit」「Firebase」の3つのサービスに依存しています。

### 外部サービス依存マップ
| サービス | 用途 | 障害時の影響範囲 | 確認先 |
| :--- | :--- | :--- | :--- |
| **Firebase** | リアルタイム通信・共有メモリ | ゲーム等の同期機能停止 | [Firebase Console](https://console.firebase.google.com/) |
| **Streamlit** | Pythonロジック実行 | 動的アプリの機能停止 | [Streamlit Dashboard](https://share.streamlit.io/) |
| **GitHub Pages** | 静的ホスティング | サイト全体の表示不可 | [GitHub Actions](https://github.com/unamuji-ai/html_utils/actions) |

### 保守方針
- **コード変更時:** 変更した機能が上記サービスのAPI仕様と整合しているかを確認する。
- **障害発生時:** まず上記「確認先」からサービス稼働状況を確認し、AIにログを提示して原因分析を依頼する。
- **ドキュメント:** 各アプリの概要は各フォルダ/ファイルのREADME.mdに記載する。

## 設計思想 (Design Philosophy)

本プロジェクトは以下の3つの原則に基づいて設計されています。

1. **ポータビリティの最大化 (Zero-Dependency Architecture)**
   - 各ツールは原則として「HTML1枚（CSS/JS含む）」で完結させる。
   - npmパッケージ管理やビルドプロセスを排除し、ファイルをアップロードするだけで即時稼働可能な状態を維持する。
   - **保守者へ:** 機能を拡張する際、外部ライブラリを追加する前に「HTML内へのCDN埋め込み」で代替できないか検討すること。

2. **疎結合なデータ連携 (Loose Coupling)**
   - ツール同士の依存関係を最小限にし、データのやり取りは `localStorage` または `Firebase Realtime DB` を介した非同期通信に限定する。
   - **保守者へ:** 特定のツールの不具合がシステム全体に波及しないよう、API連携は常に「疎結合」を維持すること。

3. **AIによるコード補完を前提とした構造 (AI-Friendly)**
   - コードはAIが理解・生成しやすいよう、冗長な抽象化を避け、可読性と自己完結性を重視する。
   - **保守者へ:** 修正を行う際は、対象ファイルのコードをそのままAIに投げ、「この設計意図を崩さずに〇〇を修正したい」と指示すること。

## 意思決定ログ (Decision Records)

- **なぜ Firebase を使うのか？:**
  - サーバーサイドの管理コストをゼロにするため。リアルタイム通信が必要なツール（[d]vote, WebRTC系）において、バックエンドを自前で立てずに同期を実現する唯一の現実解であるため。
- **なぜ Streamlit を使うのか？:**
  - データ分析やシミュレーション（モンテカルロ等）など、HTML/JSでは重い計算処理を分離するため。
- **機能拡張のルール:**
  - 新規ツールを作成した際は、必ず `history.json` にメタデータを追記し、`historyViewer.html` で検索可能にすること。

## 技術スタックの選定基準 (Tech Stack Selection)

本プロジェクトでは、ツールごとに2つの開発アプローチを使い分けています。保守者は、修正対象のファイルがどちらのカテゴリーに属するかを確認してください。

### 1. ポータブル・アプローチ (HTML/JS/CSS 1枚完結)
- **選定理由:** サーバー環境に依存せず、どこでも即座に動作させるため。また、AIがコード全体を一度に読み込んで修正・提案しやすいため。
- **対象:** 焼肉診断、ワインネーム生成、[d]vote 等の小規模ツール。
- **保守方針:** 外部ライブラリはCDN経由で読み込むこと。ビルドプロセスを介さないため、修正はブラウザでの直接確認が可能。

### 2. コンポーネント・アプローチ (TypeScript + React + Vite)
- **選定理由:** 状態管理が複雑なUI（カードバトル、対戦ロビー等）において、再利用可能なコンポーネント設計と型安全性を確保するため。
- **選定理由 (TypeScript):** 大規模なロジック修正時、型定義によって「どこを直せばいいか」を明確にし、予期せぬバグを防ぐため。
- **対象:** カードバトル、[d]vote のような複雑なUIを持つシステム。
- **保守方針:** 
  - `npm run build` によるビルドプロセスが必須。
  - 修正時は必ず TypeScript の型チェックをパスさせること。
  - **保守者へ:** 「React+Vite」のツールを修正する場合、AIに対して**「`package.json` と `src/` 配下のコンポーネントをセットで提示」**して修正を依頼すること。


## ビルドが必要なツールについて (Build-Required Projects)

本プロジェクトの大半は「HTML1枚完結」ですが、一部の高度なツール（`airhockey` 等）は TypeScript を使用しており、ビルドが必要です。

### 修正・開発手順
1. **環境構築:** `airhockey/` ディレクトリに移動し、`npm install` を実行して依存関係をインストールする。
2. **修正:** `src/main.ts` (または `main.ts`) を修正する。
3. **ビルド:** `npm run build` を実行する。
    - これにより、ブラウザで読み込み可能な `main.js` が生成されます。
4. **確認:** 生成された `main.js` が `airhockey.html` から正しく読み込まれていることを確認する。

**保守者への注意点:**
- 修正後に `main.js` をビルドし忘れると、ブラウザ上の動作に反映されません。
- `airhockey.html` を直接ブラウザで開くだけでは動作しない場合があるため、必ずローカルサーバー（Viteのプレビューモード等）を介して確認してください。
